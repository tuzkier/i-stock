# Brainstorm — 接入前的脑暴（想透"要什么"）

## 概述

brainstorm 是 **intake 之前的可选前置思考阶段**。它把一个模糊的想法 / 方向，通过**反复发散 + 收敛**想透"到底要什么"，落成一份**脑暴活文档**；只有当用户显式确认"够细了 / 可以接入了"，才把活文档冻结成脑暴方案，交给 `intake` 正式接入。

它解决的核心失败模式是 **needs 早产**：拿到一句模糊需求就冲去写契约 / PRD，把没想清的东西过早固化成承诺。brainstorm 强制在 commitment 之前**多待一会儿**——发散、追问、必要时研究，而不是一轮就收工。

三层关系：

```
discussion（要不要启动）  →  brainstorm（想透"要什么"）  →  intake（正式签约）
   现有 intake 轻对话态          本技能（可选前置）             现有正式接入
```

## 脑暴怎么做（核心方法）

脑暴做得好不好，主要看**这场对话怎么开**，不是看覆盖了多少条目。七条方法，按重要性排：

1. **一次只推进一个问题**（最重要）。每个回合只让用户拍**一个决策点**，别一口气甩一串问题让人无从答起。但"一次一个"管的是**决策点数量，不是不让你给信息**——这一个问题该带上你的分析和 2-3 个候选（见方法 3），让用户在你给的底子上拍，而不是面对空白页。脑暴是对话不是问卷，一次一个才追得深、用户也答得动。
2. **先判要不要拆 scope**（开场判一次，全程盯着）。用户说的其实是不是好几件独立的事（多个子系统 / 多条互不依赖的诉求）？是就先点破、帮他拆成几块、定先做哪块，再对那块往下脑暴——别花一堆问题去抠一个本该先拆的大需求。拆点常聊到中途才显形，**随时发现随时拆**，不只是开场判一次。
3. **带方案问，别空问**。想清一个方向时，给 2-3 个候选做法 + 各自取舍 + **你的推荐和理由**，让用户在你的判断上反应，而不是面对空白页。
4. **质疑用户给的解法**。用户提的常是解法不是问题。先逼出真问题——"你说要做 X，是不是其实想解决 Y？除了 X 有没有更直接的"——别把解法照单全收当需求。
5. **YAGNI，主动砍**。发散完要收：主动指出哪些是"这次其实不必要"的、帮用户砍掉，而不是把想到的都塞进范围。
6. **中途回述对齐**。每隔几轮，把活文档端给用户照一眼，或用一两句回述你当前的理解——"我现在理解你要的是…，对吗"——别一路闷头追问到最后才让用户看全貌。这能早抓误解、也推动收敛。
7. **节奏你定，收敛用户定**。你负责一直追到想透（别一轮就收工），但"够不够细、能不能接入"只能用户拍——见下方防早产铁律。

追问的具体手法：追到能落地（"想要更好体验"逼到"具体卡在哪、好的样子是什么"）；用真实场景 / 反例逼清边界（"举个真会用到的场景""有没有不该触发的情况"）；把你推断出的隐含假设摆出来让用户确认，而不是默默替他假设。

## 要想透什么（议程，不是问卷）

上面是"怎么聊"，这里是"聊什么"——想透"要什么"该覆盖的议程。每轮挑当前**最不确定、最影响方向**的一项，用"一次一个问题"推进；想清了就沉进活文档对应区块，没想清的留在「仍没想清」。**别按顺序一条条问。**

1. **问题本质** — 回到真问题：什么痛点 / 现状为什么不行 / 不解决会怎样（脑暴最该打磨的）
2. **目标与价值** — 完成后世界变成什么样（完成态，不是动作）；谁因此受益
3. **用户与场景** — 谁用、什么情境、典型流程、有哪些角色
4. **解法方向** — 并列几条做法 + 取舍（用方法 3：带推荐）
5. **范围与非目标** — 碰什么 / 不碰什么 / 先放后面（用方法 5：YAGNI 砍）
6. **约束 · 依赖 · 未知** — 硬约束 / 外部依赖 / 关键未知（未知 → 调研或回问用户）
7. **验收轮廓** — 怎么算做成了，关键可观察信号（轮廓级，非编号验收）
8. **风险与高代价岔路** — 选错代价高的点、危险假设，早验证或早确认

判断"想透了没"，看这八项是否大体都有谱；但**收敛仍只能由用户触发**。

## 定位与治理度

- **可选前置**：默认不改变现有入口。只有被触发（harness 建议 + 用户选，或用户显式指定）时才挂载。
- **无重治理**：不调任何 `harness ...` 控制面命令，不建 Mission / 分支 / Mission Slice / 契约，不写任何控制面文件，不调审查子 Agent，不进 Stage Gate。
- **轻治理**：当脑暴本身需要调研（外部方案可行性、既有实现、领域事实、现有代码/架构/依赖）时，**可选**调度探索 / 调研类子 Agent，但仍不写控制面，调研结论回填活文档作为发散 / 收敛的依据。按知识类型选工具：
  - **既有项目的代码事实**（现有代码怎么工作 / 架构 / 调用链 / 谁依赖谁 / 已经有什么）→ 优先用 **graphify**（`graphify-exploring` 看现状、`graphify-impact-analysis` 看影响面），用图谱取**准**知识，别靠猜或脑补。这是项目内脑暴最常用的精度来源。
  - 项目内业务 / 领域事实取证 → `discovery-analyst`；代码库关键词检索 → `Explore`；外部方案 / 行业做法 → `deep-research`。
  - graphify 不可用（未装 / 未建图 / 索引过期）时，降级为直接读码或检索，并在活文档「调研记录」注明降级——脑暴阶段不强制结构化 degradation（那是 discovery 的义务），但要让"这块知识是猜的还是查的"可见。

## 防早产铁律（不可协商）

1. **未收敛不结晶**：脑暴活文档里"目标与价值""范围与非目标""验收轮廓"三个区块，在收敛前一律标 `【未收敛】`，**不得**呈现成定稿或完整契约口径。
2. **每轮亮明未决**：每一轮回复都必须显式维护活文档的「仍没想清 / 待你拍的点」区块，并在对话里点出来。**AI 不得自行宣布脑暴完成。**
3. **收敛只能用户触发**：只有用户**显式**表达"够细了 / 可以接入了 / 就按这个接入"才触发收敛冻结。AI 不得从脑暴直接跳去写 `mission-contract.md` / PRD / 任何正式契约。
4. **只收敛"要什么"**：脑暴只把问题本质 + 目标 + 验收轮廓 + 范围与非目标想透；详细业务对象、系统场景、严谨验收场景 / 条件、技术规格**留给 intake 之后的 discovery + PRD**。脑暴不做用户画像问卷、不做编号化验收场景、不做技术方案选型决策。

## 何时使用

- 用户给的是想法 / 方向 / 痛点，但"具体要做成什么样"还没想清
- 用户显式说"先脑暴 / 先讨论想透 / 帮我把需求想清楚 / 先别急着开干 / 还没想好怎么做"
- 现有 intake 轻对话态发现"这其实需要先想透"，需要升级深度
- 需求成色不足、直接接入会让目标 / 验收早产时，由 harness 主动建议

## 何时不使用

- 需求已精确且用户带明确执行确认（"帮我实现 X / 就这么做 / 开始做 X"且 X 是清晰外部结果）→ 直接走 `intake`，不必脑暴
- 纯信息 / 闲聊 / 提问，完全不指向任何产出 → 普通对话即可，连 intake 都不进
- 已经在某个 Mission 执行中、用户消息是对当前任务的补充 → 走中途纠偏（`course-correction`），不是脑暴
- 用户只是给资料 / 让你读文档 → 那是 `source_material`，按 intake 轻对话态处理

## 活文档

- 路径：`harness-runtime/harness/brainstorms/<slug>.md`（pre-mission scratch 区；此时还没有 Mission，不污染项目仓 `docs/`）。
- `<slug>` 由需求主题派生的短横线串（如 `brainstorm-entry-rework`）；同一脑暴持续更新**同一份**文件，不每轮新建。
- 收敛冻结后，该文件作为 `source_material` 交给 intake；intake 建好 Mission 后可把它复制进 mission 目录留痕。

活文档骨架的区块名与上面**议程八项一一对应**（结构固定，内容随讨论增量生长）。**相关才填、没想到就留空或标未决**——它是思考稿不是要交付的模板，别为了填满而硬凑：

```markdown
# 脑暴：<主题>

收敛状态：【未收敛】     <!-- 收敛后改为：已收敛 · 用户确认原话："..." -->

## 需求原文 / 触发
<用户最初怎么提的，原话或贴近原话>

## 问题本质（议程1：我们到底在解决什么）
<回到真问题；区分用户给的解法 vs 底层痛点。脑暴最该被打磨的部分>

## 目标与价值【未收敛】（议程2）
<完成后世界应变成什么样的"完成态"候选，可并列多个；带"谁因此受益、怎么受益">

## 用户与场景（议程3）
<谁会用、什么情境、典型动作流、有哪些角色；轮廓级，相关才填>

## 解法方向（议程4）
<发散出的不同做法 + 各自取舍（成本/影响/风险/可行性）；带推荐；调研结论回填这里>

## 范围与非目标【未收敛】（议程5）
<大致碰什么、明确不碰什么、什么先放后面；用非目标反推意图>

## 约束 · 依赖 · 未知（议程6）
<硬约束 / 外部依赖 / 关键未知事实；未知要么调研、要么进"待你拍">

## 验收轮廓【未收敛】（议程7）
<"怎么算做成了"的轮廓性信号，非编号化验收场景 / 条件>

## 风险与高代价岔路（议程8）
<选错代价高的点、危险假设；需要早验证或早确认的>

## 仍没想清 / 待你拍的点
<每轮必须维护；这是防早产的核心区块。哪些议程项还没想清就摆在这>

## 调研记录（可选）
<若调度了调研子 Agent，结论与来源记这里>
```

## 收敛与交棒

- 收敛门：用户显式说"够细了 / 可以接入了"→ 把活文档 `收敛状态` 改为 `已收敛` 并记录用户确认原话，三个 `【未收敛】` 区块定稿。
- 收敛前自检（用户拍板后、冻结前快速扫一遍，借鉴 spec self-review）：① 有没有占位符 / TBD / 没填完的区块；② 目标·范围·验收三块互相矛盾吗；③ 有没有哪条能两种理解（有就挑一个写明）；④ scope 是不是其实该先拆没拆。发现就当场补 / 拆，不必再起一轮。
- 交棒 intake：收敛后路由到 `intake`，把活文档路径作为 `source_material` + `discussion_output` 传入；其中**用户已确认的收敛"目标与价值"**作为 `actual_task_goal` 候选（仍受 intake 的 `task_goal_fidelity` 校验），"用户确认原话"作为 `execution_confirmation`。intake 仍按其原流程建 Mission、签契约——脑暴产物是**输入**，不替代 intake。

## 与相邻阶段的边界

| 阶段 | 职责 | 与 brainstorm 的边界 |
|------|------|----------------------|
| intake 轻对话态（`discussion`） | 判断"要不要启动"，无目标时不建任务，刻意轻 | brainstorm 是它的"深度升级"：当需要把需求想透而非仅判断是否启动时进入 |
| `intake` 正式接入 | 把已想清的目标签成 Mission 契约 | brainstorm 在它之前；产出收敛物作为输入 |
| `discovery` | Mission 之后的事实取证、影响、依赖、风险 | brainstorm 不做严谨取证；只在脑暴必要时轻量调研。详细取证留给 discovery |
| `prd` | Mission 之后的严谨产品定义、业务对象、验收场景 | brainstorm 只收敛"要什么"轮廓；严谨定义全部留给 PRD，避免变成第二个 PRD |

按 `workflow.md` 执行详细步骤。
