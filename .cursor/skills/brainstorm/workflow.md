# 脑暴工作流（接入前想透"要什么"）

**Goal:** 在正式接入（intake）之前，把一个模糊想法 / 方向通过反复发散 + 收敛想透"到底要什么"，落成一份脑暴活文档；收敛后交给 intake。

**Your Role:** 你是和用户一起想问题的资深技术负责人，不是急着出活的执行手。本阶段的产物质量由"想得透不透"决定，不由"产出多快"决定。

> 防早产铁律见 `SKILL.md`，全程生效：未收敛不结晶、每轮亮明未决、收敛只能用户触发、只收敛"要什么"。

<workflow skill="brainstorm" version="1">

### 第 0 阶段 — 触发判定与进入

<step n="0" goal="判断该不该进脑暴，并让用户拍板">

判断当前需求成色：

- `precise_with_intent`：需求已是清晰外部结果且带执行确认（"帮我实现 X / 就这么做"且 X 明确）→ **不进脑暴**，直接路由 `intake`。
- `vague_or_directional`：需求是想法 / 方向 / 痛点，"具体要做成什么样"还没想清 → 候选进入脑暴。
- `explicit_brainstorm`：用户显式说"先脑暴 / 先讨论想透 / 帮我把需求想清楚 / 先别急着开干" → 直接进入脑暴。

- 条件：`vague_or_directional`（harness 判断该先想透，但用户没明说）
  用一句自然语言主动建议并让用户选，例如"这个需求我建议先一起把'要什么'想透再正式接入，还是你已经想清楚了想直接开始？"。
  - 用户选"先想透 / 先脑暴" → 进入第 1 阶段。
  - 用户选"直接开始 / 已经清楚了" → 退出本技能，路由 `intake`。
- 条件：`explicit_brainstorm`
  直接进入第 1 阶段，不再多问。
- 条件：`precise_with_intent`
  退出本技能，路由 `intake`。

本阶段**不调**任何 `harness ...` 控制面命令，不建 Mission / 分支 / Mission Slice / 契约。

</step>

### 第 1 阶段 — 初始化活文档

<step n="1" goal="建立脑暴活文档骨架">

由需求主题派生 `<slug>`（短横线串），在 `harness-runtime/harness/brainstorms/<slug>.md` 写入活文档骨架（结构见 `SKILL.md`）：

- `收敛状态` 写 `【未收敛】`。
- 「需求原文 / 触发」填用户原话或贴近原话。
- 区块名与议程八项一一对应：「问题本质」「目标与价值【未收敛】」「用户与场景」「解法方向」「范围与非目标【未收敛】」「约束·依赖·未知」「验收轮廓【未收敛】」「风险与高代价岔路」「仍没想清 / 待你拍的点」「调研记录」，先建空骨架随讨论生长。

不每轮新建文件；同一脑暴持续更新同一份。

</step>

### 第 2 阶段 — 发散 + 收敛循环

<step n="2" goal="反复打磨，直到用户认为够细">

- 循环：id=brainstorm-loop；**无轮次上限**；退出条件：用户**显式**表达收敛（"够细了 / 可以接入了 / 就按这个接入"）。AI 不得自行宣布完成。

进循环前先做一次 **scope 先判**（`SKILL.md` 方法 2）：用户说的是不是其实好几件独立的事？是就先点破、和用户一起拆成几块、定先做哪块，把拆分写进活文档，再对那块往下脑暴。**scope 不是只判一次**——拆点常聊到中途才显形，全程盯着，任何一轮发现"这其实是两件事"就当场拆。

这一阶段的实质，是用 `SKILL.md`「核心方法」开这场对话，把「议程」八项想透。**贯穿全程的硬纪律：一次只推进一个决策点，但带上你的分析和候选（方法 1）。** 每一轮：

1. **挑一项 + 带方案推进**：从议程八项里挑当前**最不确定、最影响方向**的一项（通常先打磨①问题本质——②目标⑦验收是问题想清后的产物，不是开局填出来的）。**一次只就这一项让用户拍一个决策点**；按方法 3 给 2-3 候选 + 推荐 + 理由，按方法 4 质疑用户给的解法、逼出真问题，而不是空问或一次问全八项。
2. **可选调研（轻治理）**：当某项需要外部事实才能继续时，可**按需选用**一个调研子 Agent（可选、非每轮必调），用 Task 工具调度，按知识类型选工具：
   - **既有项目的代码事实**（现有怎么工作 / 架构 / 调用链 / 依赖 / 已有什么）→ 优先 `graphify-exploring`（看现状）、`graphify-impact-analysis`（看影响面），用图谱取**准**知识，别靠猜。这是项目内脑暴最常用的精度来源（常服务议程①问题本质、⑥约束依赖、⑧风险）。
   - 项目内业务 / 领域事实 → `discovery-analyst`；代码关键词检索 → `Explore`；外部方案 / 行业做法 → `deep-research`。
   调研只为脑暴提供依据，**不写控制面、不产 discovery-brief 等正式阶段产物**；结论回填活文档「调研记录」与对应议程区块。graphify 不可用或无法调度子 Agent 时降级为主 Agent 直接读码 / 检索，并在活文档「调研记录」注明降级（脑暴不强制结构化 degradation，但要让"这块是猜的还是查的"可见）。
3. **回填 + YAGNI 砍**：把本轮想清的写进对应议程区块；按方法 5 主动指出并砍掉"这次其实不必要"的范围。目标与价值 / 范围与非目标 / 验收轮廓继续标 `【未收敛】`。相关才填，没想到的留空别硬凑。
4. **中途回述对齐 + 亮明未决**（防早产核心，方法 6）：每隔几轮把活文档端给用户照一眼、或一两句回述当前理解请用户确认；并更新「仍没想清 / 待你拍的点」，点出本轮推进了哪项、还剩哪些没想清、哪些要用户拍。用议程八项是否大体都有谱判断"想透了没"，但收敛仍只能由用户触发。

- 硬关口 `no-self-declared-convergence`：
  未收到用户显式收敛信号前，不得把 `收敛状态` 改为已收敛，不得给三个 `【未收敛】` 区块定稿，不得路由 `intake`，不得写任何契约 / PRD。

- 条件：用户提出纠偏 / 改方向
  正常吸收，回填活文档，继续循环；不视为收敛。

</step>

### 第 3 阶段 — 收敛冻结

<step n="3" goal="用户拍板后冻结脑暴方案">

- 条件：用户**显式**表达收敛
  - 收敛前自检（`SKILL.md`「收敛前自检」）：扫一遍占位符 / TBD、目标·范围·验收是否互相矛盾、有没有两可的歧义表述、scope 是否该先拆没拆；发现当场补 / 拆。
  - 把活文档 `收敛状态` 改为 `已收敛 · 用户确认原话："<用户原话>"`。
  - 三个 `【未收敛】` 区块去掉标记并定稿（目标与价值 / 范围与非目标 / 验收轮廓收敛口径）。
  - 越界复核：定稿后仍只是"要什么"轮廓，没有越界写成编号化验收场景 / 条件、业务对象模型或技术方案——若发现越界，删回轮廓级，把细化交还下游。

- 条件：用户表达"想接入但某点还没定"
  不算收敛。回第 2 阶段把该点先想清。

</step>

### 第 4 阶段 — 交棒 intake

<step n="4" goal="把收敛物交给正式接入">

宣布并路由到 `intake`，传入交接信息：

- `source_material` / `discussion_output`：脑暴活文档路径 `harness-runtime/harness/brainstorms/<slug>.md`。
- `actual_task_goal` 候选：活文档中**用户已确认的收敛"目标"**（仍由 intake 的 `task_goal_fidelity` 校验，不是免检直通）。
- `execution_confirmation`：活文档记录的用户收敛原话。
- 「范围与非目标」「验收轮廓」作为 intake 构造 `raw_intake_brief` 的参考输入，不直接当成最终范围 / 验收场景 / 条件。

交棒后由 `intake` 按其原流程建 Mission、签契约。脑暴产物是**输入**，不替代 intake 的任何关口、审查或用户确认。

</step>

</workflow>

---

## 失败路径

| 失败类型 | 触发阶段 | 恢复路径 |
|---------|-----------|---------|
| 用户其实想直接开始、不需要脑暴 | 第 0 阶段 | 退出本技能，路由 `intake` |
| 调研子 Agent 不可调度 | 第 2 阶段 | 降级主 Agent 直接检索 / 阅读，活文档注明降级；不阻断脑暴 |
| AI 想自行宣布收敛 / 直接写契约 | 第 2 / 3 阶段 | 命中 `no-self-declared-convergence`，停下，回到亮明未决、等用户显式收敛 |
| 收敛定稿越界写成 PRD / 技术方案 | 第 3 阶段 | 删回"要什么"轮廓级，把细化交还 discovery / prd |
