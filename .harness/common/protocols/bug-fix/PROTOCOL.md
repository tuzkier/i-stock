# 缺陷修复协议

缺陷修复协议通过复现、根因、补丁、回归证据和记忆更新来治理缺陷。它防止只修表象、或在没有失败信号时凭猜测修复。

## 触发条件

当用户报告缺陷、失败测试指向行为缺陷、验证发现由缺陷导致的验收场景失败、审查员指出正确性问题，或出现运行时异常 / crash 时，使用本协议。

## 控制平面绑定

| 平面 | Runtime 职责 |
|------|--------------|
| Intent | 定义现象、期望行为、影响、范围，以及当前任务的授权边界 |
| 上下文 | 载入相关模块、历史记录、测试、规格和命令约定 |
| Guide | 强制先复现、先根因后补丁、最小修复、不得弱化测试 |
| Action | 记录缺陷接入、复现、根因分析、补丁、回归验证和 harness 产物更新 |
| Feedback | 记录 red / green 证据、回归证据、验收场景 / 规格对齐情况和审查员结果 |
| Regulation | 未复现、超范围、高风险或多次修复失败时升级 |
| Memory | 决定是否更新回归测试、project-knowledge/specs、project-context或检查清单 |

## 流程

1. 缺陷接入：记录现象、影响、复现步骤或触发条件、实际结果、期望结果、疑似区域、严重级别和范围。
2. 范围与意图检查：确认缺陷属于当前范围、当前回归、验收场景阻塞项，还是后续事项。
3. 先复现：优先使用失败自动化测试，其次是可重复命令 / 脚本，再其次是 E2E / 浏览器 / 截图，最后才是最小手动步骤。没有复现证据或明确阻塞记录时不得写补丁。
4. 根因分析：用代码、测试、日志、规格或阶段文档记录直接原因、深层原因、Harness 缺口和影响半径。
5. 约束下修复：只修根因，不硬编码样例，不弱化测试，不新增未授权依赖；公共行为变化必须回到行为契约。
6. 回归包：对缺陷分类，并新增或引用能防止复发的最小回归验证。
7. 验证与审查：重跑复现、相关测试、回归包，以及可用的 lint / typecheck / build；逻辑变化时重新触发审查员检查。
8. Harness 更新：决定是否更新 project-knowledge/specs、project-context、审查检查清单、test 模板，或留下延后候选项。

## 程序化控制

- 复现命令或失败测试用于证明补丁前缺陷真实存在。
- 回归命令用于证明补丁后缺陷不再复现。
- `verify/scripts/collect_command_evidence.py` 在可用时采集最新 test / lint / typecheck / build 证据。

## 推理控制

AI / 审查员判断负责根因可信度、范围分类、影响半径、最小修复是否充分，以及阻塞复现是否可接受。

## 证据

必需证据包括：复现、red test 或阻塞 reason、green test、回归验证、范围检查、根因、审查 / verification、记忆决策。

## 禁止捷径

- 没有复现证据或阻塞复现记录时，不得写补丁。
- 不得删除、跳过或弱化失败测试。
- 不得只改输出文本来掩盖逻辑失败。
- 没有 Decision Gate 时不得扩大范围。
- 未复现的缺陷不得宣称已修复。

## 升级条件

当无法复现但用户要求继续、根因需要扩大范围、公共接口或数据模型必须变更、关键验收场景的测试环境不可用、多次修复失败，或 PRD/规格预期冲突时，必须升级。

## 记忆更新

每次缺陷修复都必须以 Harness 缺口 decision 收尾：applied、proposed 或延后。常见目标包括回归测试、project-knowledge/specs 能力规格、project-context 规则和审查员检查清单。
