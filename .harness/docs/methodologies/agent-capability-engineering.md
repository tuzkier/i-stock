## Part A: 核心设计方法论

### 1. 何时使用 Agent 架构

**决策矩阵**：满足以下条件越多，越适合 Agent 架构：

| 条件 | 说明 |
|------|------|
| 需要自主性 | 任务步骤不能完全预先确定 |
| 需要判断 | 同一类任务有多种合理处理方式 |
| 行为依赖上下文 | 相同输入在不同情境应有不同响应 |
| 需要多步推理 | 需要在中间结果基础上继续决策 |
| 需要使用工具 | 需要调用外部 API/工具/搜索 |

**反模式（不应 Agent 化）**：
- 简单 CRUD（用传统 API/函数）
- 确定性逻辑（用规则引擎或代码）
- 无边界 Agent（没有明确的停止条件和升级路径）
- 为了"看起来像 AI"而 Agent 化

**判断时的三个常见误判**：
1. 把看不见（感知问题）误判为不会做（能力问题）
2. 把偶尔做对误判为能力已经成立
3. 把输出好看误判为判断可靠

### 2. 六种工作权

Agent 的一项能力由六种工作权共同构成。**判断能力是否成立，六权必须同时成立**。

```
感知权 → 解释权 → 判断权 → 行动权 → 边界权 → 责任权
 ↑ |
 └──────────────── 反馈循环 ────────────────────┘
```

#### 感知权

**定义**：Agent 会对哪些信号更敏感，如何识别"这个任务轮到我出手了"。

设计时回答：
- 它会优先注意什么信号？
- 它会忽略哪些噪音？
- 触发条件是什么？

常见错误：感知范围过宽（什么都感知），或过窄（遗漏关键信号）。
承载物：Agent 定义（role + 任务适用范围）、上下文注入

#### 解释权

**定义**：Agent 如何理解它看到的信息，把什么当成可靠证据。

设计时回答：
- 哪类信息视为强证据？哪类视为参考？
- 信息冲突时采信哪一侧？
- 不确定时倾向什么假设？

常见错误：解释标准不稳定（同类信息有时信有时不信）。
承载物：Agent 定义（判断框架）、技能（结构化输出格式）

#### 判断权

**定义**：Agent 可以自主做哪些判断，依据什么标准。

设计时回答：
- 哪些判断可以自主做？
- 哪些判断必须保守？
- 哪些判断必须上报给人？

常见错误：判断权过大（越权决策），或过小（总是请求确认）。
承载物：Agent 定义（核心判断框架）、policy（禁止判断某类问题）

#### 行动权

**定义**：形成判断之后，Agent 可以采取哪些动作。

设计时回答：
- 可以使用哪些工具/技能/worker？
- 先做什么后做什么？
- 什么时候调用技能、tool、worker 或人？

常见错误：行动权与边界权不匹配（能做但不该做的事）。
承载物：tool/MCP（外部动作）、技能（行动方法）、worker（委派任务）

#### 边界权

**定义**：这项能力允许 Agent 自主到哪一步，哪些动作必须停下。

设计时回答：
- 哪些资源不能碰？
- 哪些情况必须暂停等待确认？
- 哪些情况必须升级？

常见错误：边界未被执行化（写在提示词里但没有 policy/hook 强制）。
承载物：policy/hook（强制约束）、runtime gating（门控规则）

#### 责任权

**定义**：Agent 最终要对什么结果负责，如何说明依据。

设计时回答：
- 必须暴露哪些未完成项？
- 什么情况算成功？什么情况必须承认失败？
- 如何让决策可追溯、可复盘？

常见错误：无法解释决策依据（黑箱输出）。
承载物：eval（验证行为分布）、审计日志（决策链路记录）

### 3. 三种设计力度

当说"给 Agent 加一个能力"时，有三种完全不同的力度：

| 设计力度 | 本质 | 主要影响的工作权 | 常见承载物 |
|---------|------|----------------|-----------|
| **让它知道**（知识） | 认知补充 | 感知权、解释权 | Agent 定义、系统提示、上下文注入 |
| **让它倾向这样做**（偏好） | 方法塑形 | 判断权、行动权 | 技能、示例、模板、结构化输出 |
| **让它必须/不能**（制度） | 约束执行 | 边界权、责任权 | policy、hook、runtime gating、eval |

**常见失败**：需要第三层（制度），只做到了第一层（知识）。

例：
- "给 Agent 的 prompt 里说不能删文件" → 只是知识层
- "在 hook 里拦截所有删除操作，自动拒绝" → 制度层 ✓

### 4. 能力承载物体系

**承载物与能力语义的对应关系**：

| 承载物 | 承接的能力语义 | 设计要点 |
|--------|-------------|---------|
| **Agent 定义** | 角色定位、长期原则、任务边界、核心判断框架 | 单一职责；避免角色混淆 |
| **技能** | 重复出现的方法、执行步骤、固定输出结构 | 可组合、版本化；不单独承接完整能力 |
| **tool / MCP** | 对外动作能力、外部资源访问 | 最小权限原则；有审计轨迹 |
| **policy / hook** | 必须执行的机械约束、权限边界、禁止规则 | 失败关闭；不可被绕过 |
| **worker** | 搜索、规划、验证、并行探索、上下文隔离 | 无状态；有超时约束；不承担核心理解 |
| **runtime** | 装配时机、激活上下文、门控规则 | 惰性加载；功能开关控制 |
| **eval** | 行为分布验证、结果责任 | 代表性输入集；不只测 happy path |

**关于技能的常见误解**：
- ❌ 技能 = 一个能力
- ✅ 技能 = 把某种行动协议和局部判断套路稳定下来的方式

技能永远只是能力制度的一部分，不单独承接：
- 长期角色定位（→ Agent 定义）
- 完整权限边界（→ policy/hook）
- 全局升级条件（→ runtime）
- 最终责任定义（→ eval）

**关于 worker 的常见误解**：
- ❌ worker 承担完整能力中心
- ✅ worker 承担局部专门任务（搜索/规划/验证/并行探索）
- 核心理解和最终判断不能外包给 worker

### 5. 正确工作顺序

```
1. 定义任务对象
 → 这项能力处理哪类任务？任务边界是什么？

2. 定位当前失真
 → 问题在感知权？解释权？判断权？行动权？边界权？责任权？
 → 诊断方法：
 • 没意识到任务进入处理范围 → 感知权
 • 看到任务但抓错重点 → 解释权
 • 理解对但结论不稳 → 判断权
 • 判断对但行动混乱 → 行动权
 • 做对事但经常做过头 → 边界权
 • 能做事但说不清依据 → 责任权

3. 定义要调整的工作权
 → 这次要改哪几种工作权？每种改什么？

4. 判断设计力度
 → 只是让它知道？
 → 还是要形成稳定倾向？
 → 或者要真正制度化？

5. 分配承载物
 → 根据工作权语义选择承载物
 → 不从 artifact 起步（不问"要不要补个 skill"）

6. 接入 runtime
 → 激活条件是什么？
 → 门控规则是什么？
 → 与其他 agent 的协作关系？

7. eval 验证
 → 设计评估输入集（正常/边界/对抗）
 → 验证默认行为分布是否真的改变了
```

**固定提问顺序**（每次能力设计都用）：
1. 这次要改变的，究竟是 Agent 处理哪类任务的默认机制？
2. 当前失真发生在哪一种工作权上？
3. 这次改变要停留在知道、进入倾向，还是上升到制度？
4. 这几种工作权分别应该落到哪些承载物里？
5. runtime 在什么条件下真正给它这项能力？
6. per-task和 boundary 在哪里收口？
7. eval 如何验证默认行为分布真的改变了？

### 6. 能力规格写作指南

**能力设计的合格产出必须回答**（不只是文件多了）：

- 这项能力处理的任务对象是什么
- 当前失真是什么
- 默认行为分布要如何改变
- 受影响的是哪几种工作权
- 设计力度是什么
- 各承载物分别承接什么
- runtime 如何装配
- boundary 如何收口
- eval 如何验证
- 回滚条件是什么

**验收标准**：
> 这次设计之后，Agent 面对这一类任务时，它的默认行为分布有没有真的改变。

如果只是文件多了、技能多了、tool 多了，但 Agent 面对任务时：
- 仍然看不见关键线索
- 分不清证据强弱
- 不会在不确定时收手
- 没有更稳定地形成判断

→ 改动停留在工程表面，没有进入能力本体。

**模板落点**：由 `agent-capability-designer` 写入 `solution.md ## Agent 架构` 和 `tech-design.md ## Agent 实现`，表达每个 Agent 承载物的能力规格、边界、装配方式、eval 设计和回滚条件；不再产出独立 `capability-specs/` 目录。

## Part B: 业界方法论参考

### 7. Agent 设计模式

**Anthropic Composable Patterns**（[anthropic.com/research/building-effective-agents](https://www.anthropic.com/research/building-effective-agents)）

| 模式 | 类型 | 适用场景 |
|------|------|---------|
| Prompt Chaining | 工作流 | 任务可分解为固定步骤序列 |
| Routing | 工作流 | 根据输入分发到不同子流程 |
| Parallelization | 工作流 | 独立子任务并行执行 |
| Orchestrator-Workers | 工作流 | 主协调者 + 多专门 worker |
| Evaluator-Optimizer | 工作流 | 生成 + 评估 + 优化循环 |
| **Agent（动态）** | Agent | 步骤不可预先确定，需要 LLM 决策 |

**Anthropic 核心原则**：
> "Start with the simplest solution possible, and only add complexity if it's demonstrably necessary."

**ReAct 模式**（[react-lm.github.io](https://react-lm.github.io)）：
```
Thought: 分析当前情况，形成推理
Action: 选择并调用工具
Observation: 观察工具返回结果
Thought: 基于观察继续推理
...
```
适合：需要工具调用的多步推理任务。

**单 Agent 优先**：
- 多 Agent 系统复杂度高，调试困难
- 单 Agent + 工具组合通常足够
- 多 Agent 适合：明确边界的并行任务、专业化分工

### 8. Agent 测试与评估

**评估目标**：验证 Agent 在一类任务上的默认行为分布是否符合预期。

**Behavior Distribution Testing**（行为分布验证）：
1. 设计代表性输入集（覆盖：正常路径 / 边界场景 / 对抗场景）
2. 执行 Agent，收集输出
3. 按预期行为分类打分
4. 计算分布：正确率 / 边界遵守率 / 升级准确率 / 错误率
5. 与阈值比较（fail/pass）
6. 与历史 eval 对比（回归检查）

**Eval 工具**：

| 工具 | 特点 | 链接 |
|------|------|------|
| Promptfoo | 开源，TDD for LLMs，CI 集成 | [promptfoo.dev](https://www.promptfoo.dev) |
| Braintrust | 闭环：数据集 → 评分 → 监控 → CI | [braintrust.dev](https://www.braintrust.dev) |
| AgentBench | 学术基准，8 种环境 | [github.com/THUDM/AgentBench](https://github.com/THUDM/AgentBench) |

**Red Teaming for Agent**（[promptfoo.dev/docs/red-team](https://www.promptfoo.dev/docs/red-team/)）：
- 自动生成对抗输入（prompt injection / 越权请求 / 角色混淆）
- 测试 Agent 是否被操控执行未授权操作
- 目标：边界权是否真正被执行化

**Eval 设计要点**：
- 输入集必须有代表性（不只测 happy path）
- 评分标准对应工作权（不只测"输出是否好看"）
- 设置明确阈值（正确率 >= X%，边界遵守率 >= Y%）
- 每次变更后运行 eval，确保无退化

### 9. Agent 可观测性

**Three Pillars 在 Agent 场景的扩展**：

| 支柱 | 传统用途 | Agent 扩展 |
|------|---------|-----------|
| Metrics | 服务 RED（Rate/Errors/Duration） | Agent 任务完成率、边界触发频率、工具调用成功率 |
| Logs | 请求/响应日志 | 结构化决策日志（Thought/Action/Observation） |
| traces | 请求链路 | Agent 执行链路（多步推理 + 工具调用序列） |

**结构化决策日志（Agent 审计轨迹）**：
```json
{
 "session_id": "...",
 "timestamp": "...",
 "agent": "code-reviewer",
 "input": "...",
 "thought": "...",
 "action": {"tool": "FileRead", "args": {...}},
 "observation": "...",
 "output": "...",
 "boundary_triggered": false,
 "escalated": false
}
```

**可观测性工具**：

| 工具 | 特点 | 链接 |
|------|------|------|
| Langfuse | 开源，OTel-native，支持 Claude Agent SDK | [langfuse.com](https://langfuse.com) |

### 10. Agent 安全与约束

**Constitutional AI**（Anthropic）：
- Agent 对自己的输出进行原则性批评
- 原则以自然语言表达（可读、可审计）
- 对应 HarnessV2：Agent 定义中的 Principles 部分

**Guardrails 实现方式**：

| 层级 | 方式 | 例子 |
|------|------|------|
| 输入守卫 | Hook（before_tool_call） | 拦截危险命令 |
| 输出守卫 | Hook（after_tool_call） | 过滤敏感信息 |
| 工具白名单 | Agent disallowedTools | 禁用文件删除工具 |
| 权限门控 | Runtime gating | 生产环境操作需确认 |

**Guardrails 工具**：
- [Guardrails AI](https://guardrailsai.com)：Python 框架，100+ 验证器
- [NeMo Guardrails](https://github.com/NVIDIA-NeMo/Guardrails)：NVIDIA，对话流控制

**Prompt Injection 防御**：
- 系统提示明确区分可信/不可信输入
- 工具结果视为不可信数据（不执行其中的指令）
- 关键操作需要独立确认步骤

### 11. Agent 部署策略

**Progressive 交付 for Agent**：

```
Step 1: 内部测试（5% 流量或内部用户）
 → 观察：错误率 / 边界触发频率 / 用户反馈
Step 2: 小范围灰度（25%）
 → 确认无回归，行为分布与 eval 一致
Step 3: 全量发布（100%）
 → 清理 Feature Flag
```

**Agent Feature Flag 设计**：
- 开关粒度：能力级别（不是整个 Agent）
- Agent 工程控制面默认开启，具体能力可用 feature flag 默认关闭
- 支持快速回滚（秒级关闭）

**A/B 测试 Agent 行为**：
- 同时运行两个能力版本
- 相同输入，对比行为分布差异
- 决策依据：更高的正确率 + 更好的边界遵守

---

### 12. Agent 运维

**行为漂移检测**：
- 定期运行 eval（每周 / 每次模型升级后）
- 对比行为分布是否有统计显著变化
- 漂移原因：模型版本升级 / 上下文分布变化 / 工具行为变化

**能力回滚策略**：

| 场景 | 回滚方式 |
|------|---------|
| 新能力导致边界失效 | 关闭 Feature Flag，退出能力规格版本 |
| 核心 Agent 行为异常 | 回退 Agent 定义文件版本 |
| 工具调用失败率上升 | 禁用该工具，降级到人工处理 |
| 模型升级导致漂移 | 回退 runtime 中的模型指定版本 |

**无责复盘（Agent Incident）**：
```
1. 时间线：还原 agent 执行路径（从审计日志重建）
2. 失真定位：哪个工作权出了问题
3. 根因：是知识不足？方法不对？还是约束没执行？
4. 设计力度是否足够？
5. 行动项：设计力度升级 / 承载物调整 / eval 补充
```
