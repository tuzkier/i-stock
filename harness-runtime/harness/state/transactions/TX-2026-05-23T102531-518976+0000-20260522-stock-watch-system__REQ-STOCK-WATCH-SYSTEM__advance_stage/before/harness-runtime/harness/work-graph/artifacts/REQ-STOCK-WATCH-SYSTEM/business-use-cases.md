# BUC 粒度规则

- 一个 BUC 必须对应用户可独立触发或可明确感知的完整业务闭环，而不是单次按钮动作。
- 一个 BUC 必须同时串起用户故事、至少一个核心 BO、至少一个状态变化或业务规则。
- 简单查看或单一切换行为保持在 Index 粒度；只有 P0/P1、高风险、多状态、异常恢复、原型驱动路径展开 Detail。
- 同一闭环内的技术实现差异不拆成独立 BUC，例如“正式来源”和“演示来源”不拆成两个能力，而在降级 BUC 中处理。
- 涉及提醒、MTS、多市场归一、本地恢复、来源降级的路径优先展开，因为这些路径会直接决定验收边界与原型范围。

# BUC Detail

## BUC-01 管理多市场自选标的

**业务目标**  
让用户把四个市场的观察对象收束到一个本地入口，同时保留市场语义和代码归一结果，避免跨市场切换时失去识别性。

**参与者**  
个人投资者

**前置条件**

- 系统处于可用状态。
- 用户正在维护本地自选清单。
- 目标市场属于第一阶段支持范围：US、HK、CN、KR。

**主成功场景**

1. 用户输入某个股票代码并声明或选择市场。
2. 系统校验市场是否受支持，并同时保留原始代码与归一代码。
3. 系统把标的加入对应市场分组的 active 自选集合。
4. 用户可在后续会话中继续看到该标的处于原分组和原排序上下文。
5. 用户把标的移出当前观察范围时，系统将其转为 archived，而不是把历史语义直接抹除。

**异常与边界**

- 市场无法识别时，不得把标的写入 active 自选集合。
- 同一市场内的重复归一代码再次加入时，系统应保持“同一观察对象”的业务语义，避免用户误以为新增了两个不同标的。
- 被归档标的重新加入观察时，应恢复为 active，而不是创建无法追溯的新对象。
- 标的归档后，绑定该标的的提醒规则进入 `suspended_by_archive` 或等价可恢复暂停状态，不再触发；标的恢复为 active 后，提醒规则按用户原先的 enabled / disabled 意图恢复。

**可观察结果**

- 用户看到标的出现在正确市场分组。
- 用户能区分原始输入与系统识别后的归一代码。
- 归档或恢复后，列表状态发生明确变化。

**关联用户故事 / BO / 规则**

- Stories: US-01, US-04
- BO: BO-001, BO-002, BO-006
- Rules: BR-001, BR-002

### GWT

**Scenario BUC-01-S01 新增多市场标的成功**

- **Given** 用户已有本地自选清单，且输入一个属于已支持市场的原始股票代码
- **When** 用户把该标的加入自选
- **Then** 系统在对应市场分组中创建 active 自选标的，并同时保存原始代码与归一代码

**Scenario BUC-01-S02 市场不可识别时拒绝进入 active**

- **Given** 用户输入的代码无法被归属到已支持市场
- **When** 用户尝试加入自选
- **Then** 系统不给该标的 active 状态，并向用户显示“当前市场无法识别或不在支持范围”的结果

**Scenario BUC-01-S03 归档后恢复**

- **Given** 某标的曾经在自选中且当前处于 archived
- **When** 用户重新把该标的纳入观察
- **Then** 系统恢复原标的为 active，并保留其市场归属与可追溯身份

### AC

- **AC-01-01** 新增成功后，标的必须进入正确的市场分组，且保留原始代码与归一代码。
- **AC-01-02** 不可识别市场的标的不得进入 active 自选集合。
- **AC-01-03** 从观察中移出时应表现为 archived 或等价可恢复状态，而不是不可恢复地丢失。
- **AC-01-04** 同一浏览器环境下重开后，已存在的 active/archived 状态仍可被识别和恢复。

**验证证据类型**

- UI 状态
- 持久化状态
- BO 状态

**Prototype Trigger**

- 多市场新增入口是否要求先选市场再输代码，还是允许输代码后自动识别。
- 归档与恢复在列表中的可见表达方式。
- 市场分组与归一代码在列表项上的信息密度。

## BUC-02 打开标的进入默认看盘布局

**业务目标**  
让用户在进入个股时无需重新配置，即可获得稳定的主图、成交量和副图观察上下文。

**参与者**  
个人投资者、技术分析用户

**前置条件**

- 至少存在一个 active 自选标的。
- 当前标的存在可展示的行情数据或可识别的降级状态。

**主成功场景**

1. 用户从自选中选中一个 active 标的。
2. 系统打开该标的的默认看盘布局。
3. 主图显示价格与默认价格类指标。
4. 成交量作为常驻副图可见。
5. 第二副图展示默认指标，并允许在候选副图间切换。

**异常与边界**

- 历史数据不足时，图表可展示，但缺失指标必须进入 partial 或 unavailable，而不是伪装为 ready。
- 默认布局必须可直接理解，不能把“需要自行完成布局设置”当成成功路径。

**可观察结果**

- 用户首次打开就能看到主图、成交量和一个副图。
- 用户能明确知道当前副图是哪一种指标。
- 局部指标不可用时，系统给出“不可解释/数据不足”的结果，而不是空白误导。

**关联用户故事 / BO / 规则**

- Stories: US-02
- BO: BO-001, BO-003, BO-004, BO-008
- Rules: BR-004, BR-006, BR-015

### GWT

**Scenario BUC-02-S01 默认布局首开成功**

- **Given** 用户选中了一个有可用行情的 active 标的
- **When** 用户进入该标的的看盘页面
- **Then** 系统默认展示主图、成交量和一个可切换副图指标

**Scenario BUC-02-S02 数据不足时局部降级**

- **Given** 当前标的的历史数据不足以支持全部默认指标
- **When** 用户进入看盘页面
- **Then** 系统仍显示可展示的价格图表，但把不可解释的指标标记为 partial 或 unavailable

### AC

- **AC-02-01** 默认布局至少包含主图、成交量常驻和一个可切换副图。
- **AC-02-02** 主图与副图必须属于同一标的上下文，不允许切换副图导致主图丢失。
- **AC-02-03** 数据不足时，页面必须明确说明哪些指标当前不可解释。

**验证证据类型**

- UI 状态
- 页面截图
- BO 状态

**Prototype Trigger**

- 主图、成交量、副图三层信息在单屏中的优先级。
- 默认副图先显示 MACD、RSI、KDJ 还是 ATR 的体验取舍。
- 数据不足提示放在图区内、图区边还是状态栏。

## BUC-04 解读 MTS 多周期趋势信号

**业务目标**  
把研究设计中的趋势、动量、波动与成交量确认收束为可解释的观察语言，帮助用户理解“为什么现在是观察、确认、强信号或风控”。

**参与者**  
研究驱动用户

**前置条件**

- 当前标的已加载到足够的历史行情，或系统已识别为数据不足。
- 当前标的已有可用的价格上下文和指标上下文。

**主成功场景**

1. 系统在进入标的或刷新数据后评估当前信号状态。
2. 系统产出趋势状态、分数带、买点类型、卖点类型、提醒等级、原因与失效条件。
3. 用户能看到 MTS 不是单一箭头，而是一组解释性结果。
4. 当触发买点时，用户能区分“趋势回调买点”与“收敛突破买点”。
5. 当出现风险或退出语义时，用户能区分“趋势破坏”“动量衰竭”“风控止损”。

**异常与边界**

- 数据不足时不得给出伪造的趋势状态或信号等级。
- 总分带只能表达排序和提醒强度，不能被包装为胜率。
- 旧复合信号不得作为对外产品语义继续暴露。

**可观察结果**

- 页面中存在明确的趋势状态、信号类型和提醒等级。
- 用户能读到触发原因或失效原因，而不是只看到抽象颜色或箭头。
- 风控出现时，页面语义明显区别于观察和确认。

**关联用户故事 / BO / 规则**

- Stories: US-03
- BO: BO-003, BO-004, BO-005
- Rules: BR-004, BR-008, BR-009, BR-010, BR-011, BR-014

### GWT

**Scenario BUC-04-S01 数据充分时输出解释性 MTS**

- **Given** 当前标的具备足够历史数据，且价格与指标上下文可解释
- **When** 系统完成一次 MTS 评估
- **Then** 页面同时展示趋势状态、分数带、信号类型、提醒等级与触发理由

**Scenario BUC-04-S02 区分两类买点**

- **Given** 当前标的已进入某类买点条件
- **When** 系统输出买点结果
- **Then** 页面必须将其标识为“趋势回调买点”或“收敛突破买点”，而不是统一为一个模糊买点

**Scenario BUC-04-S03 区分风险语义**

- **Given** 当前标的触发卖点或风控相关条件
- **When** 系统更新 MTS 结果
- **Then** 页面必须区分趋势破坏、动量衰竭和风控止损，并给出对应理由

**Scenario BUC-04-S04 数据不足时不输出伪信号**

- **Given** 当前标的历史数据不足或来源可信度不足
- **When** 系统尝试评估 MTS
- **Then** 系统只显示 data_insufficient 或等价不可解释状态，不输出观察、确认、强信号或风控结论

### AC

- **AC-04-01** MTS 展示必须同时包含趋势状态、分数带、信号类型和提醒等级。
- **AC-04-02** 买点必须至少区分趋势回调买点与收敛突破买点。
- **AC-04-03** 卖点/风险必须至少区分趋势破坏、动量衰竭、风控止损。
- **AC-04-04** 数据不足时不得显示伪造的有效信号等级。
- **AC-04-05** MTS 分数带不得被解释为收益概率或胜率表达。

**验证证据类型**

- UI 状态
- BO 状态
- 持久化状态中的最近信号快照

**Prototype Trigger**

- 趋势状态、分数带、理由、失效条件在页面上的信息层级。
- 买点/卖点/风控是否分区呈现，还是集中在同一信号卡片。
- 风控状态与强信号状态在视觉上如何避免误读。

## BUC-05 配置并接收四级提醒

**业务目标**  
让用户把价格或 MTS 语义转成可持续使用的本地提醒规则，并在同一系统里看到最近触发原因和优先级。

**参与者**  
研究驱动用户、本地使用者

**前置条件**

- 用户已选中一个 active 标的。
- 当前标的可绑定价格型或信号型提醒条件。

**主成功场景**

1. 用户为某个标的创建提醒规则。
2. 系统要求用户明确提醒类型和提醒等级语义。
3. 当价格或 MTS 条件命中时，系统触发提醒并记录触发原因。
4. 若同一时点同时命中多个提醒，系统以风控优先于观察类提醒的方式呈现。
5. 用户可停用、恢复或确认已触发提醒。

**异常与边界**

- 提醒不得只写“买入/卖出”；必须保持观察、确认、强信号、风控四级口径。
- 标的被归档后，相关提醒不得继续以 active 观察对象的方式误触发。

**可观察结果**

- 用户能看到某条提醒绑定了哪个标的、处于何种启停状态。
- 触发后能看到最近触发时间与触发原因。
- 多提醒同时命中时，用户先看到最高风险等级。

**关联用户故事 / BO / 规则**

- Stories: US-03, US-04
- BO: BO-001, BO-005, BO-006
- Rules: BR-012, BR-013

### GWT

**Scenario BUC-05-S01 创建信号型提醒**

- **Given** 用户正在查看某个 active 标的
- **When** 用户创建一条基于 MTS 的提醒规则
- **Then** 系统要求其选择观察、确认、强信号或风控等语义等级，并将规则绑定到该标的

**Scenario BUC-05-S02 多条件同时命中时优先显示风控**

- **Given** 同一标的在一次刷新中同时满足观察类条件和风控条件
- **When** 系统更新提醒状态
- **Then** 页面优先呈现风控提醒，并保留其他已命中的低优先级提醒记录

**Scenario BUC-05-S03 停用后不再继续触发**

- **Given** 某提醒规则已被用户停用
- **When** 后续数据再次满足原触发条件
- **Then** 系统不再把该规则作为 enabled 规则触发

### AC

- **AC-05-01** 提醒规则必须支持价格型和信号型两类条件。
- **AC-05-02** 提醒等级必须保持观察、确认、强信号、风控四级语义。
- **AC-05-03** 同时命中多条提醒时，风控优先级高于观察类提醒。
- **AC-05-04** 用户可识别某条提醒的启停状态、最近触发时间和触发原因。

**验证证据类型**

- UI 状态
- 持久化状态
- BO 状态

**Prototype Trigger**

- 提醒创建表单是否按“价格型 / 信号型”分流。
- 提醒列表如何表现优先级、启停状态和最近触发原因。
- 风控提醒是否需要固定在更高可见区域。

## BUC-06 重开浏览器恢复本地看盘现场

**业务目标**  
保证本地看盘体验连续，不把“重新登录或重新配置”变成用户的日常成本。

**参与者**  
本地使用者

**前置条件**

- 用户已经存在本地自选或提醒配置。

**主成功场景**

1. 用户关闭浏览器或结束会话。
2. 用户在同一浏览器配置中重新打开应用。
3. 系统恢复自选、提醒和基础看盘上下文。

**关键边界**

- 恢复是本地体验契约的一部分，不依赖账号体系。
- 恢复粒度至少覆盖自选和提醒；布局恢复策略若未定，也不能影响用户进入默认看盘路径。

### GWT

**Scenario BUC-06-S01 重开后恢复本地配置**

- **Given** 用户此前已保存自选和提醒配置
- **When** 用户关闭浏览器并重新打开应用
- **Then** 系统恢复已有自选与提醒状态，无需重新登录或重新录入

### AC

- **AC-06-01** 在同一浏览器配置内，active/archived 自选和提醒启停状态必须可恢复。
- **AC-06-02** 即使布局恢复策略未完全定稿，系统也必须能回到可用的默认看盘布局。

**验证证据类型**

- 持久化状态
- UI 状态

**Prototype Trigger**

- 首页首屏优先恢复“上次查看标的”还是“自选列表入口”。

## BUC-07 行情来源降级下继续看盘

**业务目标**  
在正式来源不可用或覆盖不足时，系统仍能维持观察连续性，但不能让用户误把演示或降级数据当成正式实时依据。

**参与者**  
个人投资者、技术分析用户、研究驱动用户

**前置条件**

- 当前系统存在至少一个来源状态可见的行情上下文。
- 用户正在查看某个标的或自选列表。

**主成功场景**

1. 系统发现当前来源不可用、覆盖不足或刷新失败。
2. 系统切换或标记为降级来源状态。
3. 页面继续展示可用的价格与图表内容。
4. 页面明确说明当前来源模式、可用性和降级原因。
5. 与该来源关联的指标、MTS、提醒同步进入相应的可信度或可解释性降级状态。

**异常与边界**

- 降级后不得继续用原先正式来源语义包装当前数据。
- 如果降级导致历史数据不足，则 MTS 与提醒必须退回到 data_insufficient 或等价状态。
- 来源变化不得改变“不做自动交易、不承诺收益、保持本地使用”的产品边界。

**可观察结果**

- 用户能看出当前是 formal、demo_fallback 或 unavailable。
- 用户能看出哪些内容仍可观察，哪些结论当前不可解释。
- 提醒和信号不会在降级场景下制造虚假的确定感。

**关联用户故事 / BO / 规则**

- Stories: US-02, US-03
- BO: BO-007, BO-003, BO-004, BO-005, BO-006
- Rules: BR-004, BR-005, BR-016

### GWT

**Scenario BUC-07-S01 正式来源失败后切入降级模式**

- **Given** 当前标的原本使用正式来源，且该来源本次刷新失败或不可用
- **When** 系统完成本次来源状态更新
- **Then** 页面显示来源已降级，并继续提供可用的图表或演示数据观察入口

**Scenario BUC-07-S02 降级导致信号不可解释**

- **Given** 当前来源已进入 demo_fallback，且支撑 MTS 的数据条件不再满足
- **When** 系统尝试更新指标、MTS 或提醒
- **Then** 系统把相关对象切换为 partial、unavailable 或 data_insufficient，而不是继续展示正式信号结论

### AC

- **AC-07-01** 来源降级后，页面必须明确当前来源模式与降级原因。
- **AC-07-02** 来源降级不应阻断基础图表观察，但必须影响相关解释对象的可信度表达。
- **AC-07-03** 若降级导致数据不足，MTS 和提醒不得继续输出看似正式的有效结论。

**验证证据类型**

- UI 状态
- BO 状态
- 日志事件

**Prototype Trigger**

- 来源状态条如何在不压缩图表可视面积的情况下持续可见。
- “可继续观察”与“当前不可解释”两类提示如何同时存在而不互相冲突。
- 降级状态在自选列表和个股详情页是否采用同一可视语义。
