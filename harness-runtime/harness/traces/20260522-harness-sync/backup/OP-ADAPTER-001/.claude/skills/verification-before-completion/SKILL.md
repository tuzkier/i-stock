# 验证 Before Completion — 完成前验证（独立技能）

## 铁律

```
没有新鲜的验证证据，不许宣称完成。
```

## 概述

这是一个可以在**任何阶段**独立调用的验证技能，不依赖于验证阶段的流程位置。

与 `verify` 技能的区别：
- `verify` 是阶段性的（code-review 后 → 交付前）
- `verification-before-completion` 是**随时可用**的（任何时候想说"完成了"之前）

## 何时使用

- 任何形式的成功/完成声明之前
- 任何满意的表达之前（"太好了！""完美！""搞定！"）
- 提交代码、创建 PR、标记任务完成之前
- 进入下一个任务之前
- sub-agent 报告成功后，独立验证

## 门控函数

```
0. 如果本次完成是针对某个问题/反馈/审查意见的修改：
 - 取出原始问题/反馈列表
 - 逐项确认：每一项是否已处理？有无遗漏？
 - 确认：修改是否引入了新问题或回归？
 （技术验证前先过这一关，发现遗漏就补上，再继续步骤 1）

1. 确定：什么命令能证明这个结论？
2. 运行：执行完整命令（重新运行，不用缓存结果）
3. 阅读：完整输出，检查退出码，统计失败数
4. 验证：输出是否支持这个结论？
 - 否 → 用证据说明实际状态
 - 是 → 带证据陈述结论
5. 只有这时：才能做出结论
```

<HARD-GATE>
跳过任何一步 = 说谎，不是验证。
修改完直接汇报 = 跳过了步骤 0。
</HARD-GATE>

## 验证对照表

| 结论 | 需要的证据 | 不够格的证据 |
|------|-----------|-------------|
| 测试通过 | 测试命令输出：0 failures | 之前的运行结果、"应该会通过" |
| Linter 无报错 | Linter 输出：0 errors | 部分检查、推断 |
| 构建成功 | 构建命令：exit 0 | linter 通过 |
| 缺陷已修复 | 测试原始症状：通过 | 代码改了，假设已修复 |
| 回归测试有效 | Red-Green 循环已验证 | 测试只通过了一次 |
| sub-agent 已完成 | VCS diff 显示变更 | Agent 报告"成功" |
| 需求已满足 | 逐项核对清单 + 可观察实际结果证据（截图 / API 响应 / CLI 输出 / 数据状态） | 测试通过 |

## 禁用词

以下词语 = 你在跳过验证：

`应该能通过` / `大概没问题` / `理论上可以` / `似乎正确` / `看起来对了` / `我觉得没问题`

## 反合理化

| 借口 | 现实 |
|------|------|
| "应该能行了" | 运行验证命令 |
| "我有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "Linter 通过了" | Linter ≠ 编译器 ≠ 测试 |
| "Agent 说成功了" | 独立验证 |
| "部分检查就够了" | 部分检查什么也证明不了 |
| "换个说法这条规则就不适用了" | 精神大于字面 |

## 关键模式

**测试：**
```
✅ [运行测试] [看到：34/34 pass] "全部测试通过"
❌ "应该能通过了"
```

**回归测试（TDD 红-绿）：**
```
✅ 编写 → 运行(通过) → 回退修复 → 运行(必须失败) → 恢复 → 运行(通过)
❌ "我写了回归测试"（没有红-绿验证）
```

**sub-agent 委派：**
```
✅ Agent 报告成功 → 检查 VCS diff → 验证变更 → 报告实际状态
❌ 信任 Agent 报告
```

## 底线

**验证没有捷径。** 运行命令。阅读输出。然后才能宣称结果。这没有商量余地。

Follow the instructions in this SKILL.md directly. No separate workflow.md needed.
