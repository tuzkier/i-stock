## 4 阶段调试流程

**每个调试 Phase 有明确的退出标准。不满足退出标准不能进入下一个 Phase。**

## 初始化

1. 读取最近执行日志条目，确认是否由其他协议 carrier 调用。
2. 如果发现 `carrier_invocation.protocol: bug-fix` 且 `to_skill: systematic-debugging`，进入 root-cause-only carrier 模式：本技能只完成 Phase 1-3，并输出 direct cause、deep cause、Harness 缺口、blast radius 和最小修复约束；不得写补丁。根因结论必须回写为 `bug-fix:root_cause` 标签事件，供 bug-fix 工作流继续消费。

<workflow skill="systematic-debugging" version="2">

<step n="1" goal="Phase 1 — 根因调查">
  - **在此阶段，禁止提出任何修复方案。只调查。**

  **1.1 完整读取错误信息**
  - 读取完整的错误信息、堆栈跟踪、日志输出
  - 注意：行号、错误码、异常类型、调用链

  **1.2 稳定复现**
  - 确定复现步骤，验证问题可以稳定触发
  - 如果不能稳定复现，添加诊断日志缩小触发条件

  **1.3 检查最近变更**
  - 运行 `git diff` 查看最近改动
  - 检查是否有：新依赖、配置变更、环境差异

  **1.4 追踪数据流**
  - 从症状出发，沿调用链向上追踪：
  1. 观察症状（什么错了）
  2. 找到直接原因（哪行代码产生了错误结果）
  3. 追问"谁调用了它，传了什么数据"
  4. 继续向上，直到找到最初的错误来源
  5. 在源头修复，不在下游打补丁

  **1.5 多组件系统特殊步骤**
  - 条件：问题涉及多个组件/服务
    - 在每个组件边界添加诊断日志
    - 运行一次，收集各边界的输入输出
    - 定位故障发生在哪两个组件之间

  **退出标准：** 能用一句话准确描述"问题发生在哪里，表现是什么"。如果说不清楚，继续调查。
</step>

<step n="2" goal="Phase 2 — 模式分析">
  - 在代码库中找到**正常工作的类似案例**
  - 逐行对比出问题的代码和正常代码
  - 记录每一个差异，无论多小
  - 理解所有隐式依赖和假设

  **退出标准：** 能列出问题代码与正常代码之间的所有关键差异。
</step>

<step n="3" goal="Phase 3 — 假设验证">
  - 明确陈述一个假设：

  > "我认为根因是 [X]，因为 [证据 Y]"

  - 设计最小验证实验（只改一个变量）
  - 运行验证

  - 条件：假设被证实
    - 进入 Phase 4

  - 条件：假设被否定
    - 记录此假设和结果
    - 基于新信息提出新假设
    - 不要在同一假设上叠加修复——回到 Phase 1 获取更多信息

  **退出标准：** 有一个被验证的根因假设，且能解释所有观察到的症状。
</step>

<step n="4" goal="Phase 4 — 实施修复">
  - 条件：当前为 root-cause-only carrier 模式
    - 停止在本技能内实施修复，将已验证根因、影响半径和最小修复约束返回给调用方。
    - 由 `bug-fix` 后续步骤编排补丁、回归验证和 Harness 缺口决策。

  **以下 4.1-4.5 仅适用于 standalone 调试或非 `bug-fix` carrier 调用。**

  **4.1 先写失败测试**
  - 创建一个测试用例，在修复前能复现问题（红）
  - 运行测试确认它确实失败

  **4.2 实施单一修复**
  - 只针对根因实施一个修复
  - 不"顺便"修其他东西

  **4.3 验证修复**
  - 运行刚才的失败测试，确认通过（绿）
  - 运行全套测试，确认无回归

  **4.4 防御纵深（修复后增加保护层）**
  根据问题的性质，考虑在以下层级增加防护：

  | 防护层 | 作用 | 示例 |
  |--------|------|------|
  | 入口校验 | 在 API/模块边界拒绝无效输入 | 参数类型检查、范围校验 |
  | 业务逻辑校验 | 确保数据在业务上合理 | 状态机转换合法性 |
  | 环境守卫 | 防止危险操作在特定上下文中发生 | 测试中禁止访问生产数据 |
  | 诊断埋点 | 为未来调试留下线索 | 关键路径的结构化日志 |

  不是每个缺陷都需要四层全加。根据风险判断。

  **4.5 修复失败升级规则**
  - 条件：修复尝试 < 3 次且失败
    - 回到 Phase 1，获取更多信息

  - 条件：修复尝试 >= 3 次且失败
    - **停止修复尝试。质疑架构。**
    - 反复修复失败通常意味着：
    - 问题不在你以为的地方
    - 现有架构无法干净地解决这个问题
    - 需要更大范围的重构而不是局部修补
    - 向用户报告情况，建议架构级讨论

  **退出标准：** 失败测试变绿，全套测试无回归，修复只针对根因。
</step>

</workflow>

## 条件等待模式（参考）

当调试涉及时序问题（flaky test、竞态条件）时，用轮询替代硬编码延时：

```
// 反模式：硬编码等待
await sleep(5000);
expect(result).toBe(expected);

// 正确模式：条件等待
await waitFor(() => getResult() === expected, '等待结果就绪', 5000);
```

硬编码延时仅在以下条件全部满足时可接受：
1. 先等待了一个触发条件
2. 等待时间基于已知的固定间隔（不是猜的）
3. 有注释说明为什么是这个时间
