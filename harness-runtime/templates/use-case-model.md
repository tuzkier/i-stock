## 模板约定

- 用例编号固定使用 `BUC-xx`（业务用例）、`SUC-xx`（系统用例）、`SUC-xx-FLOW-xx`（系统用例流步骤）、`SUC-xx-OP-xx`（系统操作）、`UIC-xx`（界面承载要求）、`DEC-xx`（待决策问题）；流步骤扇出的结局状态节点（节拍）在流步骤上附结局状态标识（如 `SUC-01-FLOW-01.empty`）。
- 严格区分三层：**SUC** = 一个参与者目标 + 系统边界内达成它的全部 flow（边界由目标决定，状态多 / 跨页 / 碰多对象都不拆 SUC）；**flow** = 一条路径（主 / 备选 / 异常）；**state** = 一个结局 / 条件；**节拍(step)** = 流步骤 × 结局状态，是覆盖颗粒。flow 与 state 多对多，严禁把“备选流”等同于“一个状态”。
- 任何空项都必须写 `不适用：原因...` 或 `待澄清：原因...`，不得留空。
- 业务用例描述业务参与者为了业务目标完成的业务过程，不等同于系统功能。
- 系统用例只来自已确认的目标系统责任；未确认责任必须进入 `## 待澄清系统责任`。
- 界面承载要求只说明原型必须承载的用户任务、信息、输入、状态、错误、权限和反馈，不决定页面数量、页面名称、布局、组件或导航。
- 用例展开中新发现的业务对象、状态、迁移、规则或术语冲突必须写入 `## 领域模型反馈`，不得在本文件里静默重写业务对象分析。

## 建模方法

| 步骤 | 方法 | 产出位置 | 阻断条件 |
|------|------|----------|----------|
| 1 | 汇总任务契约、探索简报、业务对象分析和项目知识，标注证据强度 | 输入合格性判断 | 关键目标、角色、范围或规则无来源 |
| 2 | 建立参与者-目标矩阵，区分业务参与者、系统参与者、外部系统和人工角色 | 参与者与目标矩阵 | 参与者责任混淆会改变系统边界 |
| 3 | 将业务目标组织成业务用例，写出触发、前置业务状态、主业务流程、扩展 / 异常路径和业务结果 | 业务用例模型 | 只有功能名，没有业务流程或业务结果 |
| 4 | 对每个业务用例做责任切分，明确人工活动、当前系统责任、目标系统责任、外部系统责任、不在范围责任和待澄清责任 | 系统边界与责任切分 | 目标系统责任无证据或未确认 |
| 5 | 仅从已确认目标系统责任派生系统用例，写出参与者目标、前置条件、触发、主成功流、备选流、异常流、后置结果和状态迁移 | 系统用例模型 | 系统用例来自业务动作或页面想象 |
| 6 | 把已确认系统用例流展开为系统行为描述：每个关键流步骤使用 `SUC-xx-FLOW-xx`，每个目标系统操作使用 `SUC-xx-OP-xx`，明确触发、读写对象、状态迁移、规则守卫、可观察结果和异常处理 | 系统行为描述 | 下游无法判断同一系统操作是否被交互、方案、技术设计、拆解和验证完整承载 |
| 7 | 从系统用例流步骤和系统操作推导界面承载要求，列出必须展示的信息、收集的输入、可见状态、错误、权限和反馈 | 界面承载要求 | 原型需要承载的义务无法从系统用例和系统操作推导 |
| 8 | 标记用例展开对业务对象、状态机、规则和术语的新增或修正建议 | 领域模型反馈 | 新状态 / 迁移 / 规则会改变业务语义但无法确认 |
| 9 | 建立追溯矩阵，把业务用例、系统责任、系统用例、系统行为描述、界面承载要求、业务对象、状态机、规则和验收场景连起来 | 追溯矩阵 | 关键用例或规则无法追溯到证据 |

### 结局状态节点（节拍）

> 凡一个流步骤会扇出多个用户可观察结局（状态），逐个拆成该流步骤下的具名结局节点（节拍 = 流步骤 × 结局状态），不得把多结局压进一句散文。节拍是交互 / 原型 / 验证逐结局对账的最小颗粒。节点 ID 在流步骤上附结局状态标识（如 `SUC-01-FLOW-01.empty`）；最终锚点 token 形态随原型 / lint 对账方案统一固化。
> **机器约定（扇出 token 完整性门）**：一旦某流步骤在「界面流转」表的「系统响应 / 可观察结果」用「扇出」描述多结局，就**必须**在本表为该流步骤建 ≥1 个 `SUC-xx-FLOW-xx.<state>` 节拍 token，否则散文结局会静默漏出 prototype-check 的对账分母，`harness interaction prototype-check` 报 `PRD_FANOUT_BEATS_MISSING`（FAIL）。判定信号：use-case-model 中与该流步骤同一行出现「扇出」字面量。单结局流步骤（无「扇出」描述）不在此列，不会被误伤。
